# Ревью исходных артефактов и журнал изменений

## 1. Результат

Исходные документы содержали достаточную основу для понимания модуля: назначение, пять пользовательских сценариев, 18 функциональных требований, три нефункциональных требования, диаграмму классов и набор методов frontend–backend и backend–ERP.

При этом документация была **частично пригодна к разработке**, но требовала постоянных уточнений со стороны frontend, backend и QA. Главные причины: отсутствие критериев приёмки и полной статусной модели, противоречия в API-примерах, неполное описание ошибок и неоднозначность процесса передачи заявки в ERP.

После переработки сформированы два самостоятельных артефакта:

- `FUNCTIONAL_REQUIREMENTS.md` — проверяемая бизнес- и пользовательская логика;
- `SYSTEM_SPECIFICATION.md` — согласованная системная модель и API-контракт.

## 3. Найденные проблемы

| Проблема | Риск | Исправление |
|---|---|---|
| В исходном документе заявлена статусная модель, но сама модель отсутствовала | Разработчики и QA не понимали переходы и правила агрегации | В версии 1.1 добавлена полная карта ERP → client и признаки финальных пользовательских статусов |
| Use Case `UCPARK05` использовал идентификатор другого модуля | Нарушение трассировки | Нормализован в `UC-SR-007` |
| В UC просмотра карточки повторялся шаг 2 | Двусмысленность сценария | Шаги переписаны и дополнены постусловием |
| Назначение обращения формулировалось как сообщение об ошибке, отмена и общий контакт одновременно | Неясный scope | Сформирован единый сценарий обращения в поддержку; отмена вынесена в открытый вопрос |
| ФТ были крупными и не имели критериев приёмки | Требования нельзя однозначно проверить | Требования декомпозированы, к каждому добавлены проверяемые критерии |
| «Нормальное соединение» не определено | NFR по 2 секундам нельзя воспроизвести | Требование исключено по решению владельца требований |
| Роль описана только как клиент, но спецификация содержит сотрудников | Риск неверной матрицы доступа | Пользовательский scope ограничен клиентом; сотрудники оставлены как связанные данные ERP |
| JSON-примеры содержали комментарии, пропущенные запятые, `Null`, тип `list[url]` | Примеры нельзя использовать в тестах и mock-сервисах | Подготовлены валидные обезличенные JSON-примеры |
| `equipment_id` использовался то как integer, то как внешний string | Ошибки сериализации и интеграции | Публичный API использует внутренний `asset_id: integer`; преобразование выполняет adapter |
| `service_group_id` назывался идентификатором другой сущности | Неверная фильтрация | Тип и назначение приведены к `ServiceGroup` |
| Загрузка фото описана как `photos`, но пример содержит одно поле `image` | Frontend и backend могут реализовать разные контракты | Зафиксирована загрузка одного файла за запрос и передача массива `photo_ids` в заявку |
| Endpoint поддержки не имел общего префикса API | Несогласованный routing | Нормализован в `/api/v1/support-requests` |
| В request поддержки передавалась тема, но одновременно говорилось, что тема фиксирована | Дублирование логики и риск подмены темы | Тему формирует backend по номеру заявки |
| Не было HTTP-кодов и структуры ошибок | QA не может проверить негативные сценарии | Добавлена унифицированная модель ошибок и таблица кодов |
| В диаграмме `contact` был `User`, а API возвращал ФИО, email и телефон | Несогласованная модель данных | Выделен value object `ContactInfo` |
| В разных ответах использовались `photos` и `photo_ids` | Несогласованность DTO | В ответе — `photos`, в командах создания/редактирования — `photo_ids` |
| Полный DTO многократно копировался в документе | Высокий риск расхождения при изменениях | DTO описан один раз и переиспользуется ссылочно |
| Не было связи между Use Case, ФТ и API | Сложно оценить покрытие | Добавлена таблица трассировки |
| Retry был описан текстом без критериев проверки | Разные трактовки периодов | Зафиксирована последовательность 30 сек → 1 мин → 2 мин → далее каждые 2 мин |

## 4. Что было добавлено

| Добавление | Причина | Польза |
|---|---|---|
| Цели, scope и out of scope | Исходная граница была неполной | Защищает проект от неконтролируемого расширения |
| Предусловия, альтернативные сценарии и постусловия | Сценарии описывали в основном happy path | Разработчики и QA видят исключения |
| Редактирование и удаление `DRAFT` как Use Case | Методы были в спецификации, но отсутствовали в ФТ | Согласование двух документов |
| Acceptance Criteria и Gherkin | Не было проверяемости | Прямое использование QA и на UAT |
| Бизнес-правила | Логика была распределена по разным разделам | Единая точка для спорных решений |
| Трассировка UC → FR → API | Не было контроля покрытия | Упрощает review и change impact analysis |
| Полные таблицы сущностей | В исходнике были только диаграмма и ссылки | Понимание типов, обязательности и источников |
| Валидный PlantUML | Диаграмма была только в экспортированном изображении | Возможность версионирования в Git |
| Унифицированный error envelope | Было только общее NFR | Единый контракт frontend/backend/QA |
| Реестр вопросов и решений | Пробелы не были явно отмечены | Вопросы зафиксированы, получены ответы и отражены в версии 1.1 |
| Технические рекомендации по логированию | Retry и интеграция требуют диагностики | Поддерживаемость без заявления неподтверждённых бизнес-требований |

## 5. Изменения, являющиеся техническими рекомендациями

Следующие элементы не были полностью определены в исходнике и добавлены как безопасные проектные рекомендации:

1. стандартные HTTP-коды `201`, `204`, `409`, `415`, `422`;
2. унифицированный JSON error envelope;
3. поле `request_id` для корреляции с логами;
4. формирование темы обращения на backend;
5. запрет автоматического удаления локальной заявки при расхождении с ERP;
6. использование внутреннего integer ID оборудования в публичном API;

При переносе документа в реальный проект эти решения должны быть подтверждены архитектором/backend-командой.

## 7. Изменения по ERP-спецификации

В системную спецификацию добавлены только методы, реально необходимые клиентскому модулю:

1. создание сервисной заявки в ERP;
2. получение заявок для плановой и точечной синхронизации;
3. получение справочника статусов.

Сознательно исключены ERP-методы изменения, причин отмены, связанных документов, справочника видов ремонта и оборудования, поскольку они не вызываются клиентским модулем в описанном сценарии.

### Найденные новые расхождения

| Расхождение | Риск | Решение в публичном артефакте |
|---|---|---|
| ERP POST содержит одно поле `photo`, клиент допускает 5 файлов | Неопределённый формат передачи вложений | Пробел явно зафиксирован, решение не выдумано |
| ERP POST возвращает результат и внешние реквизиты, а не полный DTO | Ошибочное ожидание полного объекта сразу после создания | Разделены создание и последующая GET-синхронизация |
| Пример ERP-справочника статусов отличается от подтверждённой карты | Некорректное отображение клиентского статуса | Приоритет отдан подтверждённой карте статусов |
| ERP описывает ошибки единым HTTP 500 + текст | Нельзя напрямую использовать как публичную error model | Backend нормализует ошибки, исходный текст клиенту не передаётся |

| Критерий | Оценка 1–5 | Комментарий |
|---|---:|---|
| Полнота функциональных требований | 5 | Все ранее заданные функциональные вопросы закрыты |
| Однозначность | 5 | Статусы, ограничения файлов, документы и права изменения определены |
| Тестируемость | 5 | Добавлены проверяемые AC для статусов, файлов, уведомлений и retry |
| Готовность frontend | 5 | Внешние контракты и пользовательские ограничения определены |
| Готовность backend | 4 | Остаётся разрыв формата передачи нескольких фото в ERP |
| Готовность QA | 5 | Есть карта статусов, негативные сценарии и error model |
| ERP-интеграция | 4 | Методы и retry описаны; формат нескольких фото требует согласования |
| NDA-safe | 5 | Названия, URL, идентификаторы и внутренние коды нормализованы |

**Итог:** комплект готов к размещению в публичном портфолио. Для production-разработки требуется закрыть только технический разрыв передачи нескольких фотографий в ERP и осознанно принять риск бесконечных retry для невосстановимых ошибок.
