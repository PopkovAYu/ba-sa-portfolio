# Функциональные требования: модуль сервисных заявок

| Параметр | Значение |
|---|---|
| Документ | Functional Requirements |
| Версия | 1.1 |
| Статус | Демонстрационный обезличенный артефакт |
| Каналы | Web / Mobile |
| Интеграция | Внешняя ERP-система |

## 1. Назначение документа

Документ описывает пользовательскую и бизнес-логику модуля сервисных заявок: создание заявки, просмотр списка и карточки, фильтрацию и поиск, взаимодействие с сервисной службой, отображение статусов и связанных документов.

Технические контракты, сущности и обработка API-ошибок приведены в `SYSTEM_SPECIFICATION.md`.

## 2. Контекст

Клиент сервисной компании должен иметь единую цифровую точку входа для регистрации проблем с оборудованием и отслеживания последующей обработки заявки. Созданная заявка передаётся во внешнюю ERP-систему асинхронно. До подтверждения передачи она хранится в локальном статусе `DRAFT`.

## 3. Цели функционала

- сократить ручные обращения в сервисную службу;
- стандартизировать данные, необходимые для регистрации заявки;
- дать клиенту актуальную информацию о заявке и её статусах;
- связать клиентское приложение с ERP без прямого доступа пользователя к ERP;
- обеспечить устойчивую передачу заявки при временной недоступности ERP.

## 4. Scope

### 4.1 Входит в scope

- создание сервисной заявки из раздела заявок;
- создание заявки из карточки оборудования с предзаполнением данных;
- выбор локации, типа сервиса и оборудования;
- создание заявки без выбора оборудования;
- добавление фотографий;
- редактирование и удаление заявки в статусе `DRAFT`;
- асинхронная передача заявки в ERP;
- просмотр списка заявок;
- фильтрация и полнотекстовый поиск;
- просмотр детальной карточки и истории статусов;
- переход к связанным документам и оборудованию;
- получение уведомлений об изменении статуса;
- отправка обращения в сервисную поддержку.

### 4.2 Не входит в scope

- назначение сервисного инженера клиентом;
- изменение заявки клиентом после выхода из статуса `DRAFT`;
- прямое редактирование данных в ERP;
- управление справочниками оборудования, локаций и сервисных групп;
- детальная логика связанных документов;
- полный жизненный цикл заявки после передачи в ERP.

### 4.3 Подтверждённые ограничения

- после успешной передачи заявки в ERP клиент не может её изменять или удалять;
- к одной заявке можно прикрепить не более 5 фотографий формата JPEG размером до 5 МБ каждая;
- обязательных каналов уведомлений нет: пользователь может отключить любое уведомление в настройках;
- связанными документами являются коммерческие предложения и отчёты о выполненных работах; они доступны в отдельном разделе «Документы»;
- финальными пользовательскими статусами являются `complete` и `canceled`;
- требование о загрузке списка за 2 секунды исключено, поскольку измеримые условия его проверки не предъявлены.

## 5. Термины

| Термин | Описание |
|---|---|
| Сервисная заявка | Обращение клиента на техническое обслуживание или ремонт оборудования. |
| Локация | Площадка клиента, доступная текущему пользователю. |
| Оборудование | Объект обслуживания, привязанный к локации. |
| ERP | Внешняя учётная система, в которой продолжается обработка заявки. |
| Черновик (`DRAFT`) | Локальная заявка, ещё не подтверждённая ERP. |
| Связанный документ | Коммерческое предложение или отчёт о выполненных работах, размещённый в разделе «Документы». |
| Активная заявка | Заявка, пользовательский статус которой не равен `complete` или `canceled`. |
| Финальная заявка | Заявка в пользовательском статусе `complete` или `canceled`. |

## 6. Роли и доступ

| Роль | Доступ | Основные действия |
|---|---|---|
| Авторизованный клиент | Доступ к заявкам по разрешённым локациям | Создание, просмотр, поиск, фильтрация, редактирование и удаление черновика, обращение в поддержку |

> Доступ других ролей в исходном пользовательском модуле не описан и не включён в этот документ.

## 7. Пользовательские сценарии

### UC-SR-001. Создание сервисной заявки

| Поле | Описание |
|---|---|
| Цель | Зарегистрировать потребность в обслуживании или ремонте оборудования. |
| Основной актор | Авторизованный клиент |
| Предусловия | Пользователь авторизован и имеет доступ хотя бы к одной локации. |
| Триггер | Пользователь выбирает действие «Создать заявку». |
| Постусловия | Заявка создана; до подтверждения ERP имеет статус `DRAFT`. |

**Основной сценарий**

1. Пользователь открывает раздел сервисных заявок.
2. Система открывает форму создания заявки.
3. Пользователь выбирает локацию.
4. Пользователь выбирает тип сервиса: техническое обслуживание или ремонт.
5. Пользователь выбирает оборудование либо оставляет поле пустым.
6. Пользователь вводит описание проблемы.
7. Пользователь при необходимости прикрепляет до 5 фотографий JPEG размером до 5 МБ каждая.
8. Пользователь отправляет форму.
9. Система валидирует данные и создаёт заявку.
10. Система показывает подтверждение создания.
11. Через 5 минут после последнего изменения система инициирует передачу заявки в ERP.
12. После успешного ответа ERP система обновляет заявку полученными данными.

**Альтернативные сценарии**

- 5a. Оборудование отсутствует в списке: пользователь оставляет поле пустым и указывает идентифицирующие сведения в описании проблемы.
- 7a. Фотографии не добавлены: заявка создаётся без вложений.
- 7b. Пользователь выбрал более 5 файлов, файл другого формата или файл размером более 5 МБ: система отклоняет недопустимые файлы и показывает причину.
- 9a. Данные не прошли проверку: система подсвечивает ошибки и не создаёт заявку до их исправления.
- 11a. ERP недоступна: заявка остаётся в статусе `DRAFT`, система выполняет повторные попытки передачи.

### UC-SR-002. Создание заявки из карточки оборудования

| Поле | Описание |
|---|---|
| Цель | Создать заявку по конкретному оборудованию с минимальным ручным вводом. |
| Основной актор | Авторизованный клиент |
| Предусловия | Пользователь открыл доступную ему карточку оборудования. |
| Триггер | Пользователь выбирает действие создания сервисной заявки. |
| Постусловия | Заявка создана с привязкой к выбранному оборудованию. |

**Основной сценарий**

1. Система открывает форму заявки.
2. Система предзаполняет локацию и оборудование.
3. Пользователь выбирает тип сервиса.
4. Пользователь вводит описание проблемы.
5. Пользователь при необходимости добавляет до 5 фотографий JPEG размером до 5 МБ каждая.
6. Пользователь отправляет форму.
7. Система создаёт заявку и запускает регламент передачи в ERP.

### UC-SR-003. Редактирование черновика

| Поле | Описание |
|---|---|
| Цель | Исправить данные до подтверждения заявки в ERP. |
| Основной актор | Авторизованный клиент |
| Предусловия | Заявка существует, доступна пользователю и имеет статус `DRAFT`. |
| Триггер | Пользователь выбирает редактирование заявки. |
| Постусловия | Изменения сохранены; пятиминутный интервал до передачи в ERP отсчитывается от последнего изменения. |

**Основной сценарий**

1. Пользователь открывает черновик.
2. Система отображает редактируемые поля.
3. Пользователь изменяет данные.
4. Система валидирует и сохраняет изменения.
5. Система переносит время отправки в ERP на 5 минут после последнего изменения.

**Исключение**

- Если заявка больше не находится в статусе `DRAFT`, система запрещает изменение и обновляет карточку актуальными данными.

### UC-SR-004. Удаление черновика

| Поле | Описание |
|---|---|
| Цель | Удалить ошибочно созданную локальную заявку. |
| Основной актор | Авторизованный клиент |
| Предусловия | Заявка доступна пользователю, имеет статус `DRAFT` и ещё не подтверждена ERP. |
| Триггер | Пользователь подтверждает удаление заявки. |
| Постусловия | Заявка удалена и отсутствует в списке. |

### UC-SR-005. Просмотр и фильтрация списка

| Поле | Описание |
|---|---|
| Цель | Найти нужную заявку и оценить её текущее состояние. |
| Основной актор | Авторизованный клиент |
| Предусловия | Пользователь авторизован. |
| Триггер | Пользователь открывает раздел сервисных заявок. |
| Постусловия | Отображён список, соответствующий выбранным условиям. |

**Основной сценарий**

1. Система показывает активные заявки по доступным локациям.
2. Пользователь применяет один или несколько фильтров либо вводит поисковый запрос.
3. Система обновляет список.
4. Пользователь открывает выбранную заявку.

**Альтернативный сценарий**

- Результатов нет: система отображает empty state и сохраняет выбранные фильтры.

### UC-SR-006. Просмотр карточки заявки

| Поле | Описание |
|---|---|
| Цель | Получить полную актуальную информацию по заявке. |
| Основной актор | Авторизованный клиент |
| Предусловия | Заявка существует и доступна пользователю. |
| Триггер | Пользователь выбирает заявку в списке. |
| Постусловия | Пользователь видит карточку и историю статусов. |

**Основной сценарий**

1. Система загружает карточку заявки.
2. Система показывает доступные данные заявки, фотографии и текущий статус.
3. Система показывает историю изменения статусов в виде таймлайна.
4. При наличии внешнего номера система предоставляет переход в раздел «Документы», где размещаются связанные коммерческие предложения и отчёты о выполненных работах.
5. При наличии оборудования система позволяет открыть его карточку.

### UC-SR-007. Обращение в сервисную поддержку

| Поле | Описание |
|---|---|
| Цель | Передать вопрос или сообщение по заявке сервисной службе. |
| Основной актор | Авторизованный клиент |
| Предусловия | Пользователь находится в карточке заявки или в разделе сервисных обращений. |
| Триггер | Пользователь выбирает действие «Связаться с сервисом». |
| Постусловия | Обращение направлено в поддержку, пользователю отправлено подтверждение. |

**Основной сценарий**

1. Система открывает форму обратной связи.
2. Пользователь вводит сообщение.
3. Система проверяет длину сообщения.
4. Система отправляет обращение в сервисную службу с привязкой к заявке.
5. Система отправляет пользователю уведомление о получении обращения.

## 8. Статусная модель

В публичной версии внутренние идентификаторы ERP нормализованы. Пользователь видит агрегированный клиентский статус: несколько внутренних или ERP-статусов могут отображаться одинаково.

| Внутреннее состояние | Статус ERP (label) | ERP code (обезличено) | Статус клиента | Client code | Финальный для клиента |
|---|---|---|---|---|---|
| `DRAFT` | — | — | Черновик | `draft` | Нет |
| `CREATED` | Создано | `ERP_CREATED` | Создан | `created` | Нет |
| `RECEIVED_FROM_CLIENT` | Получено от клиента | `ERP_RECEIVED_FROM_CLIENT` | Создан | `created` | Нет |
| `PROCESSING` | Отправлено в работу | `ERP_SENT_TO_WORK` | Обрабатывается | `processing` | Нет |
| `COMPONENTS_AWAIT` | Ожидает поставки запчастей | `ERP_WAITING_FOR_PARTS` | Ожидает поставки запчастей | `components_await` | Нет |
| `ENGINEER_ASSIGNED` | — | — | Назначен инженер | `engineer_assigned` | Нет |
| `ENGINEER_ARRIVED` | — | — | Инженер прибыл | `engineer_arrived` | Нет |
| `IN_PROGRESS` | Выполняется | `ERP_IN_PROGRESS` | Выполнен | `complete` | Да |
| `HAS_SERVICE_REPORT` | Оформлен отчёт о выполненных работах | `ERP_REPORT_CREATED` | Выполнен | `complete` | Да |
| `COMPLETE` | Выполнено | `ERP_COMPLETED` | Выполнен | `complete` | Да |
| `CANCELED` | Отменено без выполнения | `ERP_CANCELED` | Отменен | `canceled` | Да |
| `ENGINEER_TRANSFER` | Передать другому инженеру | `ERP_TRANSFER_ENGINEER` | Назначен инженер | `engineer_assigned` | Нет |
| `PLANNED` | Запланировано | `ERP_PLANNED` | Обрабатывается | `processing` | Нет |

**Правила:**

1. По умолчанию список содержит все заявки, кроме заявок с клиентскими кодами `complete` и `canceled`.
2. Завершённые и отменённые заявки доступны через фильтр.
3. После выхода заявки из локального статуса `draft` клиент не может изменять или удалять её.
4. История статусов отображает пользовательские статусы после применения маппинга ERP → client.

## 9. Функциональные требования

| ID | Требование | Критерии приёмки |
|---|---|---|
| FR-SR-001 | Система должна позволять авторизованному клиенту создавать сервисную заявку. | После отправки валидной формы создаётся одна заявка, доступная в списке пользователя. |
| FR-SR-002 | При создании заявки пользователь должен выбрать доступную ему локацию. | В списке выбора отображаются только разрешённые пользователю локации; без локации форма не отправляется. |
| FR-SR-003 | Пользователь должен выбрать тип сервиса: техническое обслуживание или ремонт. | Поле обязательно; выбранное значение сохраняется и отображается в карточке. |
| FR-SR-004 | Пользователь должен иметь возможность выбрать оборудование, относящееся к выбранной локации. | После смены локации список оборудования обновляется; недоступное оборудование выбрать нельзя. |
| FR-SR-005 | Система должна позволять создать заявку без выбора оборудования. | При пустом поле оборудования форма может быть отправлена; в карточке отсутствие оборудования отображается явно. |
| FR-SR-006 | Пользователь должен указать описание проблемы. | Пустое описание не принимается; введённый текст сохраняется без потери данных. |
| FR-SR-007 | Система должна позволять прикрепить к заявке не более 5 фотографий JPEG размером до 5 МБ каждая. | Допустимые файлы загружаются; шестой файл, файл больше 5 МБ или неподдерживаемый формат отклоняется с понятным сообщением. |
| FR-SR-008 | При создании заявки из карточки оборудования система должна предзаполнить локацию и оборудование. | Открытая форма содержит данные выбранного оборудования; пользователь не вводит их повторно. |
| FR-SR-009 | Новая заявка должна создаваться в локальном статусе `draft` до подтверждения ERP. | До успешного ответа ERP карточка показывает статус «Черновик» и допускает редактирование. |
| FR-SR-010 | Пользователь должен иметь возможность редактировать и удалять заявку только в статусе `draft`. | Изменение или удаление черновика выполняется; действие для любого другого статуса отклоняется. |
| FR-SR-011 | Система должна отправлять заявку в ERP через 5 минут после последнего изменения. | Отправка не начинается ранее 5 минут; новое изменение черновика перезапускает отсчёт. |
| FR-SR-012 | При неуспешной передаче в ERP система должна повторять отправку до успеха. | Попытки выполняются через 30 секунд, затем через 1 минуту, затем через 2 минуты и далее каждые 2 минуты; заявка не теряется. |
| FR-SR-013 | При каждой повторной отправке backend должен передавать в ERP один и тот же стабильный внешний идентификатор операции. | Все retry одной заявки содержат одинаковый `external_id`; повторная обработка не должна создавать новую локальную заявку. |
| FR-SR-014 | После успешного ответа ERP система должна сохранить внешний идентификатор и номер заявки. | Поля внешнего идентификатора и номера соответствуют ответу ERP; заявка становится недоступной для изменения клиентом. |
| FR-SR-015 | По умолчанию система должна отображать активные заявки пользователя. | При первом открытии не отображаются заявки с клиентскими кодами `complete` и `canceled`; они доступны через фильтр. |
| FR-SR-016 | Список должен отображать номер, статус, тип сервиса, оборудование, серийный номер и источник заявки. | Для каждой строки отображаются доступные значения; отсутствующие необязательные значения показаны единообразно. |
| FR-SR-017 | Система должна поддерживать фильтрацию по статусу, оборудованию, локации, типу сервиса и отсутствию оборудования. | Комбинация фильтров возвращает только соответствующие заявки; фильтры можно сбросить. |
| FR-SR-018 | Система должна поддерживать регистронезависимый поиск по статусу, номеру, оборудованию, серийному номеру и типу сервиса. | Одинаковый запрос в разных регистрах возвращает одинаковый набор результатов. |
| FR-SR-019 | При отсутствии результатов система должна отображать empty state. | Вместо пустого списка показывается сообщение об отсутствии заявок; ошибки загрузки не маскируются под empty state. |
| FR-SR-020 | Система должна отображать детальную карточку заявки. | Карточка содержит доступные поля из системной спецификации и соответствует выбранной заявке. |
| FR-SR-021 | Система должна отображать текущий клиентский статус и историю его изменений. | Статус определяется по карте ERP → client; история упорядочена по времени. |
| FR-SR-022 | При наличии внешнего номера система должна предоставлять переход в раздел «Документы». | Переход открывает раздел, в котором размещаются связанные коммерческие предложения и отчёты о выполненных работах. |
| FR-SR-023 | Если у заявки отсутствует внешний номер, переход в раздел «Документы» должен быть недоступен с сообщением «Документы отсутствуют». | Пользователь не может выполнить переход и видит указанное сообщение. |
| FR-SR-024 | При наличии оборудования система должна позволять перейти в его карточку. | Переход открывает оборудование, связанное с заявкой; при отсутствии оборудования действие скрыто или недоступно. |
| FR-SR-025 | При изменении статуса система должна сформировать пользовательское уведомление с учётом настроек пользователя. | При разрешённом уведомлении создаётся одно уведомление с идентификатором заявки; при отключённом уведомлении оно не отправляется. Обязательных каналов нет. |
| FR-SR-026 | Система должна позволять отправить сообщение в сервисную поддержку с привязкой к заявке. | Валидное сообщение направляется в поддержку; пользователь получает подтверждение. |
| FR-SR-027 | Тему обращения должна формировать система по стандартному шаблону. | Пользователь не вводит и не изменяет тему; отдельные категории обращений не используются. |
| FR-SR-028 | Пользователь должен видеть только заявки по доступным ему локациям. | Запрос списка и карточки не возвращает данные недоступной локации. |

## 10. Нефункциональные требования

| ID | Категория | Требование | Критерии приёмки |
|---|---|---|---|
| NFR-SR-001 | Безопасность | Методы модуля должны быть доступны только авторизованным пользователям. | Запрос без действительного токена отклоняется; доступ к заявке чужой локации запрещён. |
| NFR-SR-002 | Надёжность | Временная недоступность ERP не должна приводить к потере заявки. | Заявка хранится локально, а повторные попытки выполняются по установленному расписанию до успеха. |
| NFR-SR-003 | Идемпотентность интеграции | Повторная передача одной заявки в ERP должна использовать один и тот же стабильный `external_id`. | Во всех retry передаётся одинаковое значение; frontend ключ идемпотентности не формирует и не передаёт. |
| NFR-SR-004 | Согласованность | Результаты синхронизации ERP должны одинаково отражаться в списке, карточке и истории статусов. | После успешной синхронизации все представления используют одну актуальную версию данных. |
| NFR-SR-005 | UX | Empty state должен отличаться от состояния технической ошибки. | Пользователь различает отсутствие заявок и сбой загрузки. |
| NFR-SR-006 | Ошибки | Ошибки должны отображаться в унифицированном пользовательском формате. | Сообщение содержит понятный текст и не раскрывает stack trace или внутренние сведения ERP. |
| NFR-SR-007 | Файлы | Допускается не более 5 фотографий JPEG размером до 5 МБ каждая. | Ограничения проверяются до привязки файлов к заявке; нарушение возвращает ошибку валидации. |
| NFR-SR-008 | Синхронизация | Backend должен получать изменения заявок из ERP не реже одного раза в 15 минут. | Изменение ERP появляется в приложении после очередного успешного цикла синхронизации. |

## 12. Acceptance Criteria в формате Gherkin

### AC-SR-001. Создание заявки с оборудованием

```gherkin
Given авторизованный клиент открыл форму создания заявки
And выбрал доступную локацию
And выбрал оборудование этой локации
And выбрал тип сервиса
And указал описание проблемы
When клиент отправляет форму
Then система создаёт одну заявку в статусе draft
And отображает подтверждение создания
And заявка появляется в списке клиента
```

### AC-SR-002. Создание заявки без оборудования

```gherkin
Given авторизованный клиент заполнил обязательные поля формы
And не выбрал оборудование
When клиент отправляет форму
Then система создаёт заявку без оборудования
And интеграционный адаптер подготавливает служебное значение для ERP
And в клиентской карточке оборудование отображается как отсутствующее
```

### AC-SR-003. Валидация обязательных полей

```gherkin
Given пользователь открыл форму создания заявки
When пользователь отправляет форму без локации, типа сервиса или описания проблемы
Then система не создаёт заявку
And показывает ошибку возле каждого незаполненного обязательного поля
```

### AC-SR-004. Ограничение количества фотографий

```gherkin
Given к заявке уже добавлено 5 фотографий
When пользователь пытается добавить шестую фотографию
Then система не загружает файл
And сообщает о максимальном количестве 5 фотографий
```

### AC-SR-005. Ограничение размера и формата фотографии

```gherkin
Given пользователь выбрал файл размером более 5 МБ или файл не в формате JPEG
When система проверяет файл
Then файл не сохраняется
And пользователь видит причину отклонения
```

### AC-SR-006. Редактирование черновика

```gherkin
Given заявка находится в статусе draft
When клиент изменяет описание и сохраняет заявку
Then система сохраняет новое описание
And время плановой передачи в ERP устанавливается на 5 минут после сохранения
```

### AC-SR-007. Запрет изменения после передачи

```gherkin
Given заявка вышла из статуса draft
When клиент пытается изменить или удалить заявку
Then система отклоняет действие
And показывает актуальное состояние заявки
```

### AC-SR-008. Повторная отправка в ERP

```gherkin
Given срок передачи заявки наступил
And ERP временно недоступна
When первая попытка завершается ошибкой
Then backend повторяет отправку через 30 секунд
And затем через 1 минуту
And затем через 2 минуты
And далее каждые 2 минуты до успешной передачи
```

### AC-SR-009. Стабильный ключ интеграции

```gherkin
Given backend поставил заявку в очередь передачи в ERP
When выполняются первая и повторные попытки передачи
Then каждая попытка содержит одинаковый external_id
And frontend не передаёт ключ идемпотентности
```

### AC-SR-010. Фильтрация активных заявок

```gherkin
Given в системе есть заявки с разными клиентскими статусами
When пользователь впервые открывает список
Then система не показывает заявки со статусами complete и canceled
And пользователь может включить их через фильтр
```

### AC-SR-011. Фильтрация и поиск

```gherkin
Given в списке есть заявки разных локаций, статусов и типов сервиса
When пользователь выбирает несколько фильтров и вводит поисковую строку
Then система возвращает только заявки, соответствующие всем активным условиям
And регистр поисковой строки не влияет на результат
```

### AC-SR-012. Empty state

```gherkin
Given по выбранным фильтрам нет заявок
When система успешно завершает запрос списка
Then система показывает empty state
And не показывает сообщение о технической ошибке
```

### AC-SR-013. Переход к документам

```gherkin
Given заявке присвоен внешний номер
When пользователь выбирает действие перехода к документам
Then система открывает раздел Документы
And раздел может содержать коммерческие предложения и отчёты о выполненных работах
```

### AC-SR-014. Документы недоступны до присвоения номера

```gherkin
Given у заявки отсутствует внешний номер
When пользователь открывает её карточку
Then переход в раздел Документы недоступен
And система показывает сообщение "Документы отсутствуют"
```

### AC-SR-015. Настройки уведомлений

```gherkin
Given пользователь отключил уведомление об изменении статуса
When статус заявки изменяется
Then система не отправляет отключённое уведомление
```

### AC-SR-016. Удаление черновика

```gherkin
Given заявка находится в статусе draft
When клиент подтверждает удаление
Then система удаляет заявку
And заявка больше не отображается в списке
```

### AC-SR-017. Запрет доступа к чужой локации

```gherkin
Given заявка относится к локации, недоступной текущему клиенту
When клиент запрашивает карточку заявки по идентификатору
Then система не возвращает данные заявки
And отвечает унифицированной ошибкой доступа или отсутствия ресурса
```

### AC-SR-018. Отправка сообщения в поддержку

```gherkin
Given клиент открыл форму обращения по заявке
And ввёл сообщение длиной не более 500 символов
When клиент отправляет форму
Then система направляет обращение в сервисную службу
And формирует тему по стандартному шаблону
And отправляет клиенту подтверждение получения
```
